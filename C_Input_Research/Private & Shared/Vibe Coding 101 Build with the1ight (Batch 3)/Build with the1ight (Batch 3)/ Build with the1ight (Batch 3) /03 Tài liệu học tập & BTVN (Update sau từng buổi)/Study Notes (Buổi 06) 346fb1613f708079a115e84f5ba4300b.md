# Study Notes (Buổi 06)

# 📝 Ghép Công Cụ: Từ PRD đến Deploy App Live

## 🎯 1. Tóm tắt chung (Executive Summary)

Buổi học số 06 thuộc khóa học "Vibe Coding 101" tập trung vào quy trình ghép nối các thành phần Frontend (FE), Backend (BE), Database (DB) và API để tạo ra một ứng dụng hoàn chỉnh có URL công khai. Mục tiêu chính là hướng dẫn những người không chuyên về kỹ thuật (non-tech) cách sử dụng hệ sinh thái công cụ AI và nền tảng đám mây để hiện thực hóa sản phẩm từ bản đặc tả (PRD). Nội dung bao gồm các bước từ thiết kế giao diện, thiết lập cơ sở dữ liệu, quản lý mã nguồn đến triển khai bảo mật trên môi trường thực tế.

## 💡 2. Các điểm nhấn quan trọng (Key Takeaways)

- **Tách biệt UI và Logic:** Sử dụng Google Stitch cho giao diện tĩnh (mockup) và Google AI Studio để tạo logic Frontend giúp kiểm soát sản phẩm tốt hơn.
- **Cơ sở dữ liệu là "Ký ức":** Supabase đóng vai trò lưu trữ dữ liệu lâu dài, được thiết lập thông qua sơ đồ ERD và mã SQL do AI hỗ trợ tạo ra từ PRD.
- **Git/Github là cầu nối:** Đây không chỉ là nơi lưu trữ mã nguồn mà còn là nguồn cấp dữ liệu cho quá trình triển khai tự động.
- **Bảo mật là ưu tiên hàng đầu:** Việc quản lý API Key qua biến môi trường (Environment Variables) và file `.gitignore` là bắt buộc để tránh rò rỉ thông tin và rủi ro tài chính.
- **Mô hình Nhạc trưởng:** Người làm sản phẩm đóng vai trò nhạc trưởng điều phối AI và các công cụ bổ trợ thay vì trực tiếp viết mã phức tạp.

## 📖 3. Nội dung chi tiết

### 3.1. Từ Giao diện (UI) đến Logic Giao diện (UI Logic)

Quy trình được khuyến nghị cho người mới bắt đầu để tránh việc AI "đoán mò" và giúp dễ dàng sửa đổi từng màn hình:

- **Google Stitch:** Công cụ tạo UI mockup từ prompt. Kết quả là bản thiết kế tĩnh, đẹp nhưng chưa có tính tương tác logic.
- **Google AI Studio:** Nhận đầu vào là PRD và thiết kế từ Stitch để tạo ra mã Frontend có logic. Kết quả là ứng dụng có thể chạy ngay trên trình duyệt với đầy đủ tương tác và trạng thái (state).
- **Quy trình chuẩn:** 📄 PRD + 🎨 Stitch → 💻 AI Studio → Code chạy được.

### 3.2. Kết nối Database để tạo Full MVP

Để ứng dụng có khả năng lưu trữ dữ liệu sau khi làm mới trình duyệt, cần thiết lập Database (DB):

- **ERD (Entity Relationship Diagram):** AI sẽ đọc PRD để vẽ bản đồ quan hệ dữ liệu (ví dụ: Users, Posts, Comments).
- **Supabase:** Nền tảng cung cấp Postgres, Auth, Storage và API.
    - **Chạy SQL:** AI tạo mã `CREATE TABLE` từ ERD, người dùng dán vào SQL Editor trên Supabase để tạo bảng.
    - **Kết nối FE với Supabase:** Cần hai thông tin từ mục Settings > API của Supabase là **Project URL** (địa chỉ dự án) và **API Key (Anon)** (chìa khóa xác thực).

### 3.3. Quản lý mã nguồn với Git và Github

Git đóng vai trò như "lịch sử Google Docs" cho mã nguồn, giúp lưu trữ và phục hồi các phiên bản:

- **4 Khái niệm cốt lõi:**
    - *Repo:* Thư mục được theo dõi.
    - *Commit:* Ảnh chụp trạng thái code tại một thời điểm.
    - *Branch:* Nhánh phát triển song song.
    - *Merge:* Gộp các nhánh về nhánh chính (main).
- **Luồng làm việc:** Tải code từ AI Studio → Tạo Repo trên Github → Tải code lên (thủ công hoặc qua Github Desktop) → Kiểm tra mã nguồn trên mây.

### 3.4. Triển khai ứng dụng (Deploy) với Vercel

Vercel là nền tảng lưu trữ (host) miễn phí cho Frontend với cơ chế "Push to deploy":

- **Cơ chế:** Khi liên kết Github với Vercel, mỗi lần người dùng đẩy code mới lên Github, Vercel sẽ tự động cập nhật ứng dụng.
- **Preview Deployment:** Mỗi nhánh (branch) sẽ có một URL riêng để thử nghiệm tính năng mới trước khi đưa vào bản chính thức.
- **Tên miền:** Mặc định có dạng `your-app.vercel.app`, nhưng có thể cấu hình tên miền riêng dễ dàng.

### 3.5. Bảo mật và Quản lý Chìa khóa (Keys)

Đây là phần cực kỳ quan trọng để tránh việc lộ thông tin dẫn đến mất tiền (ví dụ: bot quét key để chạy tiền ảo):

- **Phân loại Key:**
    - *Anon Key:* Chìa khóa công cộng, an toàn khi dùng ở FE nếu đã bật RLS (Row Level Security).
    - *Service Role Key:* Quyền quản trị tối cao, **tuyệt đối không** đưa vào FE hoặc đẩy lên Github.
- **Quản lý đúng cách:**
    - Không viết trực tiếp key vào code (hardcode).
    - Sử dụng biến môi trường: File `.env` ở máy cục bộ và thiết lập trong Dashboard của Vercel khi chạy chính thức.
    - Sử dụng `.gitignore` để chặn các file nhạy cảm (`.env`, `node_modules`) không bị đẩy lên Github.

### 3.6. Tổng kết Hành trình Sản phẩm

Sản phẩm hoàn chỉnh là sự phối hợp giữa ba nhân tố:

1. **Con người:** Viết PRD, review thiết kế, kết nối key và kiểm tra bảo mật.
2. **AI:** Tạo UI, ERD, SQL và logic Frontend.
3. **Công cụ:** Supabase (data), Github (code), Vercel (deploy), `.env` (bảo mật).

## 🚀 4. Hành động cần làm (Action Items)

- [ ]  Kiểm tra DB Supabase đã có các bảng đúng theo ERD và có dữ liệu thực tế.
- [ ]  Rà soát Github repo, đảm bảo không bị lộ bất kỳ API Key nào.
- [ ]  Xác nhận file `.gitignore` hoạt động đúng và file `.env` không nằm trên Github.
- [ ]  Thiết lập đầy đủ các biến môi trường (Environment Variables) trên Vercel Dashboard.
